# MiroFish Full-Stack i18n — Design Spec

## Problem

MiroFish was just fully translated from Chinese to English. The upstream maintainer (666ghj/MiroFish) wants to offer both languages. Currently there's no i18n infrastructure — all strings are hardcoded English.

## Goal

Add bilingual (English / Chinese) support across all three layers: frontend UI, backend API messages, and LLM prompts. Users select language via a UI toggle that defaults to browser preference.

## Architecture

```
User clicks language toggle (or browser default)
  |
  v
Frontend: vue-i18n reads from en.json / zh-CN.json
  |
  v
Axios interceptor sets Accept-Language header on all API requests
  |
  v
Backend middleware: reads Accept-Language, sets g.locale
  |
  +---> API handlers: msg(key, g.locale) for error/status strings
  |
  +---> LLM services: load_prompt(name, locale) for prompt templates
  |       (locale passed explicitly — g.locale captured before thread spawn)
  |
  +---> Background threads: locale captured from g.locale before spawn,
        passed as parameter through the call chain
```

## Supported Locales

- `en` — English (current default)
- `zh-CN` — Simplified Chinese (reconstructed from git diff)

## Layer 1: Frontend (vue-i18n)

### Dependencies

- `vue-i18n@9` (Vue 3 compatible)

### File Structure

```
frontend/src/i18n/
  index.js          # vue-i18n setup, locale detection, export
  en.json           # English strings (~600 keys)
  zh-CN.json        # Chinese strings (~600 keys)
```

### vue-i18n Setup

```js
import { createI18n } from 'vue-i18n'
import en from './en.json'
import zhCN from './zh-CN.json'

function detectLocale() {
  return localStorage.getItem('locale')
    || (navigator.language.startsWith('zh') ? 'zh-CN' : 'en')
}

const i18n = createI18n({
  legacy: false,
  locale: detectLocale(),
  fallbackLocale: 'en',
  messages: { en, 'zh-CN': zhCN }
})

export default i18n
```

Key points:
- `fallbackLocale: 'en'` ensures missing `zh-CN` keys fall back to English (not raw key names)
- `legacy: false` enables Composition API (`useI18n()`)
- `i18n.global.locale` is reactive — components re-render on locale change

### Integration

1. `main.js`: `import i18n from './i18n'` then `app.use(i18n)`
2. All Vue templates: replace inline strings with `{{ $t('key') }}` or `:placeholder="$t('key')"`
3. JS/computed properties: use `t('key')` with `const { t } = useI18n()` or `i18n.global.t('key')`

### Language Switcher

The project has no shared navbar component. `Home.vue` has its own nav, and `MainView.vue` has a separate one. The language toggle needs to be placed in both:

- **Home.vue navbar**: Add a globe icon + dropdown ("English" / "中文") in the top nav
- **MainView.vue sidebar**: Add the same toggle in the sidebar header
- On change: `i18n.global.locale.value = newLocale`, `localStorage.setItem('locale', newLocale)`
- On load: `detectLocale()` in `i18n/index.js` handles this

The store at `frontend/src/store/` only contains `pendingUpload.js` (not Vuex/Pinia). `i18n.global.locale` is already reactive, so no additional store integration is needed.

### Axios Interceptor

In `frontend/src/api/index.js`, add to the existing request interceptor:

```js
import i18n from '@/i18n'
// In the request interceptor:
config.headers['Accept-Language'] = i18n.global.locale.value
```

This sends a simplified locale value (`en` or `zh-CN`), not a full RFC 7231 `Accept-Language` header. The backend middleware is designed to match this format.

### Generating zh-CN.json

Use git diff to reconstruct Chinese strings:
1. Checkout the pre-translation state of each Vue file (commit `1536a79`, the last commit before our translation)
2. Extract Chinese strings from templates
3. Map them to the same keys used in `en.json`
4. This can be done semi-automatically with a script or manually for accuracy

## Layer 2: Backend API Messages

### File Structure

```
backend/app/i18n/
  __init__.py       # get_locale(), msg() functions
  en.json           # English API messages
  zh-CN.json        # Chinese API messages
```

### Message Catalog Format

```json
{
  "project_not_found": "Project not found: {project_id}",
  "zep_api_key_missing": "ZEP_API_KEY is not configured",
  "graph_not_built": "Graph not built yet. Please call /api/graph/build first.",
  "entity_not_found": "Entity not found",
  "simulation_not_ready": "Simulation is not ready",
  "report_generating": "Report is being generated..."
}
```

Messages support `{placeholder}` substitution via Python's `str.format_map()`.

### Middleware

In `backend/app/__init__.py`, add before_request handler:

```python
@app.before_request
def set_locale():
    from flask import g, request
    import os
    default = os.environ.get('APP_LANGUAGE', 'en')
    accept = request.headers.get('Accept-Language', default)
    g.locale = 'zh-CN' if accept.startswith('zh') else 'en'
```

Note: This expects the simplified locale format sent by the MiroFish frontend (`en` or `zh-CN`), not the full RFC 7231 `Accept-Language` with quality values. Direct API consumers should send `Accept-Language: zh-CN` (not `zh-CN,zh;q=0.9`).

### Usage in API Handlers

```python
from app.i18n import msg

# Before:
jsonify({"success": False, "error": "Project not found"})

# After:
jsonify({"success": False, "error": msg('project_not_found', project_id=project_id)})
```

The `msg()` function reads `g.locale` internally, falls back to English for missing keys.

### Scope

Only user-facing API response messages need translation. Internal log messages, comments, and docstrings stay English. Also includes `INTERVIEW_PROMPT_PREFIX` in `simulation.py` (a user-facing string injected into agent interviews).

Estimated: ~50-80 unique message keys across the 3 API blueprint files.

### load_prompt Implementation

```python
import os

PROMPTS_DIR = os.path.join(os.path.dirname(__file__))

def load_prompt(name: str, locale: str = 'en') -> str:
    """Load a prompt template file for the given locale.

    Returns the raw template string with {placeholder} markers.
    Callers must call .format() or .format_map() to substitute values.
    Falls back to English if the requested locale file is missing.
    """
    path = os.path.join(PROMPTS_DIR, locale, f'{name}.txt')
    if not os.path.exists(path):
        path = os.path.join(PROMPTS_DIR, 'en', f'{name}.txt')
    with open(path, 'r', encoding='utf-8') as f:
        return f.read()
```

Key design decisions:
- `load_prompt` does NOT read `g.locale` — it takes `locale` as an explicit parameter. This is because long-running operations (report generation, simulation prep) run in background threads where Flask's request context (`g`) is unavailable.
- Returns a template string with `{placeholder}` markers. Callers substitute with `.format()` or `.format_map()`.
- All existing prompt constants are **removed** from Python source files and moved to `prompts/{locale}/` text files. No constants remain in the service files.

### Background Thread Locale Propagation

Several API handlers spawn background threads:
- `build_graph()` in `graph.py`
- `prepare_simulation()` in `simulation.py`
- `generate_report()` in `report.py`

Flask's `g.locale` does not exist in background threads. The pattern:

```python
# In the API handler (request context available):
locale = g.locale

# Pass to background function:
thread = Thread(target=do_work, args=(locale, ...))

# In the background function:
def do_work(locale, ...):
    prompt = load_prompt('report_plan_system', locale)
```

This requires adding a `locale` parameter to:
- `GraphBuilder.__init__()` or `build()`
- `OasisProfileGenerator` methods
- `SimulationConfigGenerator` methods
- `ReportAgent.__init__()`
- `SimulationRunner` methods that use prompts

## Configuration

### .env.example

Update `.env.example` comments to English (they were in Chinese). Add:

```
# Default language (en or zh-CN). Users can override via UI toggle.
APP_LANGUAGE=en
```

This sets the fallback when no `Accept-Language` header is present (e.g., direct API calls).

## Implementation Order

1. **Backend utilities**: Create `backend/app/i18n/__init__.py` (msg function) and `backend/app/prompts/__init__.py` (load_prompt function). Add locale middleware to `__init__.py`.
2. **Backend prompt extraction**: Extract all prompt constants from the 5 service files into `prompts/en/` text files. Refactor services to use `load_prompt()`. Add `locale` parameter to classes/methods.
3. **Backend API messages**: Extract error/status strings from the 3 API blueprint files into `i18n/en.json`. Refactor handlers to use `msg()`.
4. **Frontend vue-i18n setup**: Install vue-i18n, create `i18n/index.js`, add Axios interceptor.
5. **Frontend string extraction**: Extract all inline strings from Vue files into `en.json`, replace with `$t()` calls. Add language switcher UI.
6. **Chinese translation files**: Generate `zh-CN.json` (frontend), `zh-CN.json` (backend messages), and `prompts/zh-CN/*.txt` from git history.
7. **Integration testing**: Verify all 7 verification items.

---

## Constraints

- No additional runtime dependencies beyond `vue-i18n@9`
- Backend i18n is a thin layer (JSON files + one utility function), no heavy framework
- Prompt files use `{placeholder}` syntax compatible with Python `str.format()`
- English is always the fallback locale
- Locale detection order: `localStorage` > `navigator.language` > `APP_LANGUAGE` env var > `'en'`

## Verification

1. Switch to Chinese in UI toggle — all frontend text displays in Chinese
2. API error responses return Chinese messages when `Accept-Language: zh-CN`
3. LLM output is in Chinese when locale is `zh-CN` (test with ontology generation)
4. Switch back to English — everything reverts
5. Fresh browser with Chinese `navigator.language` defaults to Chinese
6. Verify ReACT loop parsing (`Final Answer:`, `<tool_call>` tags) works correctly when LLM responds in Chinese — the parsing logic must be language-agnostic
7. Frontend build succeeds
8. Python syntax check passes
